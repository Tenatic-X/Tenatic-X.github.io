04. Transfer Learning with TensorFlow Part 1: Feature Extraction¶

We've built many CNN models from scratch that all learnt, however this is done through adding more laers, changing learning rate, adjusting every parameters etc.

This is very time consuming, so we use a different method called transfer learning. In other words, using a model's learnt patterns (aka. weights) from a problem to then be applied on our own problem. Two main benefits are:

  1. Can leverage an existing neural network architevture proven to work on problems similar to our own.
  2. Can leverage a working neural network architecture which has already learned patterns on similar data to our own. Achieving greater results with less custom data.

In other terms, we can utilise models that have similar problems to ours, rather than hand-crafting our own neural network architectures and building them from scratch.

Instead of training from scratch, we take the patterns a model has learned from datasets like ImageNet, then use them as foundation of our own.

What we're covering¶

  • Introduce transfer learning (a way to beat all of our old self-built models)
  • Using a smaller dataset to experiment faster (10% of training samples of 10 classes of food)
  • Build a transfer learning feature extraction model using TensorFlow Hub
  • Introduce the TensorBoard callback to track model training results
  • Compare model results using TensorBoard

Transfer learning with TensorFlow Hub: Getting great results with 10% of the data¶

Many problems that you'll require deep learning for, would've most likely been solved with a similar model. And these are often available from TensorFlow Hub, aka Kaggle.

Its a repository for existing model components, making it so you can import and use fully trained models quickly.

Surprisingly, transfer learning is very efficient. Where the model can learn from 10% of the data, and outperform the best model that we can create from scratch.

We will see this in practice by training on 10% of training data from the 10_food_classes dataset.

Example of what we're building. Taking pre-trained model and adding our custom layer on top. (Top layer is our input layer of our problem. In this case, the 10 food category).

import tensorflow as tf

BATCH_SIZE = 32

test_dir='10_food_classes_10_percent/test/'

datagen=ImageDataGenerator(rescale=1/255.)

train_data_10_percent=datagen.flow_from_directory(train_dir,
                                                  target_size=IMAGE_SHAPE,
                                                  batch_size=BATCH_SIZE,

print('Testing images:')
test_data=datagen.flow_from_directory(test_dir,

Setting up callbacks (things to run whilst our model trains)¶

Callbacks is a very important concept in the future, as we further build more experiments.

Its an extra functionality thats added to your models, where it performs during or after training. A few popular callbacks include:

  • Experiment tracking with TensorBoard - log the performance of multiple models, then view and compare them visually using TensorBoard. Its a dashboard for inspecting neural network parameters. This is helpful in comparing results of different models on your data.
  • Model checkpointing - save your model in the middle of training, so you can come back to it from where it was last trained. Helpful when training takes forever and can't be reasonably done in one sitting.
  • Early stopping - leave your model training for an arbitrarily long time, then let it stop early when it detects no improvement. Helpful with large datasets where you don't know how long training will take.

These will be explored eventually, though our first focus will be on TensorBoard callback, and how it can be used.

TensorBoard can be accessed with tf.keras.callbacks.TensorBoard().

It's main function is saving a model's training performance metrics to a specified log_dir.

By default, logs are recorded every epoch using the update_freq='epoch' parameter. Which is good, as you can record logs by batchs or integer, which is much more frequent, and slows everything down. Epoch gives enough idea on model progress, and also don't slow down by spending lots of time writing logs.

To track our modelling experiments with TensorBoard, we'll create a function which creates a TensorBoard callback for us.

Note: Each model needs their own TensorBoard callback instance. Which is why we will form a function to streamline the process for future models. This is because, they work like logs. if the same log is reused on a different model, it overwrites whatever that was archived in it.

In [3]:
# create tensorboard callback (functionized due to every model needing its own callback)
import datetime
def create_tensorboard_callback(dir_name, experiment_name):
    log_dir = dir_name + "/" + experiment_name + "/" + datetime.datetime.now().strftime('%Y%m%d-%H%M%S')
    tensorboard_callback = tf.keras.callbacks.TensorBoard(
        log_dir=log_dir)
    
    print(f'Saving TensorBoard log files to: {log_dir}')
    return tensorboard_callback

As we're running multiple experiments, its best to track them in some way, so you don't lost track.

Our function saves a model's performance log to directories that are names accordingly: [dir_name]/[experiment_name]/[current_timestamp]

  • [dir_name] - the overall log directory
  • [experiment_name] - the particular experiment
  • [current_timestamp] - the time the experiment started, based on when datetime code was ran

Note: This can be one way of keeping track of logs, and its naming method. You are able to modify the naming and placing of where the logs are to suit your needs, or your project. Make sure names are understandable to you and your team.

Question: There are many models after refining. How do I know which is best?

Usually, try find the model that has the lowest error rate overall on a benchmark problem. For example, we're working with images. So we'll try find models which perform best on ImageNet: https://huggingface.co/papers/trending

  1. Once you select a model you like say this efficient net model, go to Model Variation and select a model. Good rule of thumb is the higher numbers say B6 compared to B0, means a better model. But it does often translate to longer training time.

  2. Click the b0-feature-vector and click the download button

Question: Why are we downloading feature vector? Shouldn't it be classification because we're doing an image classification problem???

This is where different types of transfer learning come into play, as in the feature extraction and fine tuning.

  1. "As is" transfer learning: Where you take a pretrained model, and apply it 'as is' onto your problem without any changes.

    • An example is most computer vision models are pretrained on ImageNet, which contained 1000 different classes. So when passing a single image, it'll produce 1000 prediction probabilities. One for every class it's studied.
      • This helpful if you have 1000 classes, and they happen to be the exact same classes as was used in ImageNet. Though is not as useful, if you have 10 classes of food variation.

  2. Feature extraction transfer learning: Taking the underlying presets (weights) of a pretrained model, and adjust its outputs to suit your problem.

    • A model is pretrained with many layers, but the top layer outputs 1000 different classes. Which doesn't suit your 10 class food problem. To adjust this, you may remove the original activation layer and replace it with your own that has the right number of output classes. Most important takeaway is the top layer is changed to suit your problem. But everything else underneath is frozen in place.
      • This way, all underlying patterns and weights remain the same as original. Which is good if your data is not exactly the same as the problem's original data, but similar enough.

  3. Fine-tuning transfer learning: Taking the layers of the pretrained model, and adjusting the weights to fit your problem even better.

    • This means you train on some, most, or even all of the layers. It's useful for large datasets with many classes, where the classes don't fit in so cleanly with the original classes and data, the pretrained model used.

Common workflow in fine tuning, is only train the top 2-3 layers, while the bottom layers are all frozen. Then slowly trickles down the stack of layers.

After training the 2-3 layers, you unfreeze the next 2-3 layers, and repeat the training process, to further fine tune.

Question: Why only train 2-3 layers at once?

The lower the layers in a model, the closer it is to the input layer, and the more larger features it learns. E.g. layer 100 may learn the general body shape of a dog, while layer 2 learns the creases of the dog's leg. You'll want the larger features (main details) to remain as is, since these can help differentiate between a cat and a dog. While further fine-tuning on the specific details between a cat and dog, can help further distinguish apart, compared to simple shapes.

def create_model(model_url, num_classes=10):
    '''Takes a tensorflow hub URL and creates a Keras Sequential model with it

    Args:
        model_url (str): A tensorflow hub feature extraction URL 
        num_classes (int): Number of output neurons in output layer,
            should be equal to number of target classes, default 10.

    Returns:
    An uncompiled Keras Sequential model with model_url as feature
    extractor layer and Dense output layer with num_classes outputs.
    '''
    # download pretrained model and save it as keras layer
    feature_extractor_layer = hub.KerasLayer(model_url,
                                             trainable=False, # freeze underlying layers
                                             name='feature_extraction_layer',
                                             input_shape=IMAGE_SHAPE+(3,)) # define input image shape. (3,) > is the number of colour channels. We want to add it to input as its also part of our data with the colour channels
    
    # create our model
    model = tf.keras.Sequential([
        tf.keras.layers.Lambda(lambda x: feature_extractor_layer(x)), # use the feature extraction layer as the base
        tf.keras.layers.Dense(num_classes, activation='softmax', name='output_layer') # creating our own output layer
    ])

    return model

import matplotlib.pyplot as plt

# plot the validation and training data separately
def plot_loss_curves(history):
    '''
    returns separate loss curves for training and validation metrics
    '''
    loss = history.history['loss']
    val_loss = history.history['val_loss']

    accuracy = history.history['accuracy']
    val_accuracy = history.history['val_accuracy']

    epochs = range(len(history.history['loss']))

    # plot loss
    plt.plot(epochs,loss,label='training_loss')
    plt.plot(epochs,val_loss,label='val_loss')
    plt.title('Loss')
    plt.xlabel('Epochs')
    plt.legend()

    # plot accuracy
    plt.figure()
    plt.plot(epochs,accuracy,label='training_accuracy')
    plt.plot(epochs,val_accuracy,label='val_accuracy')
    plt.title('Accuracy')
    plt.xlabel('Epochs')
    plt.legend()

Comparing models using TensorBoard¶

We've visually compared the performance of the two models via the accuracy and loss scores/graphs. But what happens if you have more than 2 models?

The environment TensorBoard comes in handy for this visualization.

Since we used tensorboard callbacks, all training logs are saved and can be uploaded into TensorBoard.dev

Once you upload the modelling experiments onto TensorBoard.dev, you can share it to others, accompanying the notebook.

Uploading experiments to TensorBoard¶

To upload the TensorFlow logs onto TensorBoard, you can use the command:

code
Upload TensorBoard dev records

!tensorboard dev upload --logdir ./tensorflow_hub/ \
  --name "EfficientNetB0 vs. ResNet50V2" \ 
  --description "Comparing two different TF Hub feature extraction models architectures using 10% of training images" \ 
  --one_shot

Where:

  • --logdir is the target upload directory
  • --name is the name of the experiment
  • --description is a breif description of the experiment
  • --one_shot exits the TensorBoard uploader once uploading is finished

Running tensorboard dev upload command will ask you to authorize upload to TensorBoard.dev. After you authorized the upload, log files will then be uploaded.
